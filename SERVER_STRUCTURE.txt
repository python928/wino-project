Important backend notes
-----------------------
- Active settings entrypoint: `app-backend/backend/settings.py`
- Active root router: `app-backend/backend/urls.py`
- `common/` is not a major business app, but it is used by middleware/imports.
- Root short-link routes live in backend root:
  - `/s/{id}/`
  - `/p/{id}/`
- Active notifications device path is routed through:
  - `/api/notifications/devices/`

Backend module roles
--------------------
- `users/`
  - auth, profile, OTP, followers, store reports, system settings, trust settings, abuse flags
- `catalog/`
  - categories, products, packs, promotions, reviews, favorites, product reports
- `ads/`
  - sponsored campaigns and click tracking
- `notifications/`
  - notification list/read flows + FCM device routes
- `subscriptions/`
  - plans, payment requests, merchant subscriptions, dashboard
- `wallet/`
  - coin packs, purchases, balances, approvals
- `analytics/`
  - recommendations, events, trust signals, interest profiles, retention cleanup
- `feedback/`
  - user feedback and admin review flow

Platform scope
--------------
- Android platform folder is retained.
- iOS/Web/macOS/Linux/Windows folders were removed from the repo.
- Some internal names still use legacy identifiers such as `dzlocal_shop`, but the current product-facing brand is `Wino`.

Key architecture notes
----------------------
- `Store == User` is the core backend domain rule.
- The Flutter app uses named routes via `RouteGenerator`.
- Localization is split between ARB-generated strings and runtime translations.
- Wino branding is visible in `WinoApp` and the Android notification channel (`wino_channel`).
- The repo now contains both technical docs and memoire-oriented docs at the root.

Recent structure / product evolution to remember
------------------------------------------------
- first-run language onboarding is now a real product flow, not just a translation setting.
- location permission education is centralized in shared Flutter helper code.
- merchant ads dashboard now includes stronger explainability text.
- empty placeholder directories under `the_app/lib/features/` were cleaned up.
- API docs should no longer mention the removed duplicate devices route.
- auth flow structure is now partially centralized through `the_app/lib/presentation/auth/widgets/auth_flow_components.dart`.
- category picking became a shared, multi-context primitive instead of a one-off screen behavior.
