# ما الذي ينقص المشروع الآن؟ (تقييم عملي محدث)
آخر مراجعة: 23-03-2026

## 1) نواقص حرجة قبل النشر
- إعدادات الإنتاج ما تزال غير آمنة في `backend/settings.py`:
  - `DEBUG=True`
  - `ALLOWED_HOSTS=["*"]`
  - `CORS_ALLOW_ALL_ORIGINS=True`
  - `SECRET_KEY` ثابت داخل الكود
- لا يوجد فصل قوي بين إعدادات dev/prod (ملف إعداد إنتاج مستقل + env validation إلزامي).
- لا توجد مراقبة أخطاء إنتاجية (Sentry/بديل) ولا trace/APM لزمن استجابة API.
- لا يوجد runbook واضح للطوارئ (rotate keys, revoke tokens, restore DB).

## 2) ما تم تحسينه مؤخرا (بالمقارنة مع المراجعات السابقة)
- تم إضافة CI فعلي في `.github/workflows/ci.yml`:
  - `python manage.py check`
  - `flutter analyze`
- إضافة منظومة `wallet` (شراء/موافقة على شراء العملات).
- إضافة منظومة `ads` كـ app مستقل بدل الاعتماد على promotions فقط.
- إضافة `SystemSettings` في admin للتحكم بـ:
  - عملات التسجيل الأول
  - حد عملات تسجيل الدخول اليومي
- إضافة top-up يومي للعملات في `/api/users/me/` كل 24 ساعة عند النزول تحت الحد.

## 3) نواقص مهمة في الجودة والاتساق
- تغطية الاختبارات ما تزال أقل من المطلوب مقارنة بحجم الميزات الجديدة (ads/wallet/subscriptions).
- توجد احتمالية عدم اتساق بين app config والـ backend routes:
  - التطبيق يستهلك مسارات اشتراك مثل `/plans/` و`/payment-requests/`
  - لكن `subscriptions/urls.py` الحالي لا يعرّفها بعد
- كثرة ملفات patch/debug المؤقتة في repo الجذر داخل `app-backend/` و`the_app/` تحتاج تنظيف.
- تحتاج مراجعة migration history خاصة users/ads للتأكد من ترتيب نظيف قبل release.

## 4) نواقص في تجربة المستخدم (جاهزية إنتاج)
- تحسين رسائل الخطأ في حالات OTP (rate limit / expired / wrong code) لرفع الوضوح UX.
- استكمال empty/loading/error states في بعض شاشات wallet/ads dashboard.
- اختبار شامل لتزامن الرصيد (wallet + daily top-up + posting costs) عبر كل مسارات الاستخدام.

## 5) نواقص في التحليلات والتوصية
- النظام جيد تشغيليا لكنه يحتاج إطار قياس رسمي:
  - Precision@K / Recall@K / NDCG@K
  - KPI dashboard لتأثير ads على CTR/Conversion
- فصل أوضح بين events التي تأتي من التطبيق وevents المشتقة server-side للتتبع والتحليل.

## 6) أولويات التنفيذ المقترحة (مرتبة)
1. إغلاق ثغرات إعدادات الإنتاج والأسرار (أولوية قصوى).
2. إصلاح/توحيد routes الاشتراكات بين backend وFlutter config.
3. توسيع الاختبارات الحرجة:
   - Auth + OTP + JWT refresh
   - Wallet buy/approve
   - Ads create/list/click + limits
   - Subscription access-status/dashboard
4. إضافة مراقبة أخطاء وأداء (Sentry + basic APM/log correlation ids).
5. تنظيف ملفات patch المؤقتة قبل أي tag/release.

## 7) تعريف "جاهز للنشر" (Definition of Done)
- إعدادات الأمن الإنتاجية مفعلة بالكامل عبر env.
- CI يمر بالكامل في كل Push/PR.
- مسارات التطبيق والباك اند متطابقة ومثبتة باختبارات تكامل.
- منظومة wallet/ads/subscription مدعومة باختبارات حرجة.
- مراقبة الأخطاء والأداء فعّالة.
- خطة نسخ احتياطي واسترجاع مجربة فعليا.
