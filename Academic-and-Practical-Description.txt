# التوصيف الاكاديمي والتقني لمشروع Wino

آخر تحديث: 23-03-2026

## 1) التعريف بالمشروع
Wino منصة تجارة محلية رقمية (Local O2O Marketplace) تربط بين:
- تطبيق Flutter (Android-first حاليا).
- Django + DRF API للخادم.

المبدأ البنيوي المركزي: **Store == User**.
كل مستخدم يمكن تمثيله كتاجر بدون نموذج Store منفصل.

---

## 2) الاشكالية البحثية
كيف نبني نظام بحث وتوصية محلي قابل للتفسير ويعمل بكفاءة ضمن:
- قيود الموقع الجغرافي الواقعي.
- تفاوت جودة المتاجر والثقة.
- ضغط الأداء مع نمو البيانات.
- الحاجة لطبقة monetization عملية للتجار (Ads/Wallet/Subscriptions).

---

## 3) الاهداف العلمية والتقنية
1. تصميم معمارية معيارية تفصل Presentation / API / Domain Logic.
2. تحسين البحث المحلي (Nearby + ولاية + ترتيب متعدد العوامل).
3. بناء توصية سلوكية هجينة مدعومة بـ InteractionLog.
4. تفعيل تحليلات تشغيلية قابلة للتوسع (events + trust signals).
5. دمج monetization عملي: ads + wallet + subscription access.

---

## 4) المعمارية الحالية
### 4.1 Backend (Django/DRF)
- Django 5.0.6 + DRF 3.15.2
- JWT عبر SimpleJWT
- Apps الفعلية:
  - users
  - catalog
  - ads
  - notifications
  - subscriptions
  - wallet
  - analytics
  - feedback
  - common (middleware)

### 4.2 Frontend (Flutter)
- Provider/ChangeNotifier
- RouteGenerator + named routes (go_router موجود كاعتماد فقط)
- Firebase Messaging + Local Notifications
- Deep links عبر `app_links`
- QR share/scan عبر `qr_flutter` + `mobile_scanner`
- Android-only platforms في المشروع حاليا (تم حذف ios/web/macos/linux/windows)

### 4.3 Integration Layer
- REST APIs تحت `/api/...`
- Media upload (صور منتجات/حملات/دفعات/feedback)
- FCM device registration

---

## 5) النموذج المفاهيمي للبيانات
### 5.1 كيانات رئيسية
- User (حساب + ملف متجر + حقول توثيق)
- SystemSettings (first_login_coins / daily_login_coins)
- TrustSettings + AbuseFlag
- Category, Product, ProductImage
- Pack, PackProduct, PackImage
- Promotion, PromotionImage
- Review, Favorite, ProductReport, StoreReport
- AdCampaign
- SubscriptionPlan, MerchantSubscription, SubscriptionPaymentRequest
- CoinPurchase, CoinTransaction, CoinPackPlan
- InteractionLog, UserInterestProfile
- Feedback

### 5.2 قيود اتساق مهمة
- Review/Favorite uniqueness بقيود منطقية على مستوى المستخدم/الهدف.
- Ownership checks في create/update/delete.
- Anti-abuse throttling + cooldowns لبلاغات/مراجعات/OTP.
- `SystemSettings` و`TrustSettings` كنماذج singleton تشغيلية.

---

## 6) منطق ذكي منفذ فعليا
1. توصية هجينة عبر `analytics/recommendations.py`.
2. تتبع تفاعل عبر `/api/analytics/events/`.
3. Trust signals عبر `/api/analytics/trust-signals/`.
4. تقييم مصداقية البلاغات/المراجعات (trust scoring).
5. شارات تحقق متجر (`is_verified`) مع تدفقات تقييم أهلية.
6. Daily coin top-up في مسار profile حسب SystemSettings.

---

## 7) الواجهات والروابط التشغيلية
- Auth: `/api/auth/token/`, `/api/auth/token/refresh/`
- Users: `/api/users/...`
- Catalog: `/api/catalog/...`
- Ads: `/api/ads/campaigns/`
- Analytics: `/api/analytics/recommendations/`, `/api/analytics/events/`, `/api/analytics/trust-signals/`
- Feedback: `/api/feedback/`, `/api/feedback/my/`
- Subscriptions: `/api/subscriptions/plans/`, `/api/subscriptions/payment-requests/`, `/api/subscriptions/merchant-subscriptions/...`
- Wallet: `/api/wallet/...` مع approve/reject لطلبات الشراء
- Short links: `/s/{id}/` و`/p/{id}/` مع redirect إلى scheme `wino://...`

---

## 8) الحوكمة والامن
1. JWT rotation + blacklist.
2. DRF permissions افتراضي IsAuthenticated.
3. Scoped throttling (OTP/reports/reviews/analytics/feedback/payment).
4. Request correlation id عبر `common.middleware.RequestIdMiddleware`.

ملاحظة:
- الإعدادات أصبحت env-driven، لكن القيم الافتراضية لا تزال development-friendly، وبالتالي يلزم production hardening قبل الإطلاق.

---

## 9) KPIs مقترحة للتقييم الاكاديمي/العملي
1. CTR/Conversion للتوصيات والاعلانات.
2. زمن الاستجابة (P95/P99) لمسارات search/recommendation.
3. Retention للتاجر والمستخدم.
4. نسبة نجاح تدفقات wallet/subscription.
5. جودة الثقة: نسبة البلاغات منخفضة المصداقية مقابل البلاغات المفيدة.

---

## 10) حدود النسخة الحالية
1. النظام rule/score-based أكثر من كونه Learning-to-Rank كامل.
2. لا تزال المراقبة الإنتاجية (Sentry/APM) غير مفعلة بالكامل.
3. ترجمة الواجهة تحسنت (AR/EN/FR + locale switch) لكنها تحتاج تعميم كامل على كل النصوص.

---

## 11) افاق التطوير البحثي
1. Learning-to-rank / ranking calibration.
2. Explainability dashboard للتوصيات والبلاغات.
3. A/B testing منهجي على search/recommendation/ads.
4. طبقة قياس أعمال للتاجر (funnel + cohort).

---

## 12) خلاصة
Wino حاليا يمثل منصة O2O محلية ناضجة مع:
- نواة domain متماسكة (Store==User).
- بحث/توصية/ثقة/Monetization في منظومة واحدة.
- قابلية تطوير عالية نحو نسخة إنتاج كاملة للسوق الجزائري.
